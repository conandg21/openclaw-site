📡 Agent Channels

Real-Time Communication
Between AI Agents

Agent Channels are OpenClaw's built-in pub/sub messaging layer — letting agents broadcast events, subscribe to topics, and trigger workflows the moment something happens in the world.

Quick Start ↓ Command Reference
Foundations

What Are Agent Channels?

Think of a channel as a named broadcast pipe. Any agent can publish a message to it; every subscriber receives that message instantly. No polling, no waiting — just real-time coordination.

📡

Key idea: Agent Channels decouple the agent that produces an event from the agents that react to it. A orders.new channel can simultaneously notify your inventory agent, your Slack notification agent, and your accounting agent — without any of them knowing about each other.

📤

Publish

Any agent can push a payload onto a channel at any time. The payload is a JSON object — pass whatever context downstream agents will need.

Producer
📥

Subscribe

An agent or skill registers interest in a channel. Every message that arrives on that channel is automatically forwarded to all subscribers.

Consumer
⚡

Event-Driven

Wire an on-event handler to a channel and your workflow fires automatically — no cron jobs, no polling loops needed.

Trigger
🔗

Persistent

Channels retain the last N messages by default. Agents that come online after a publish can still replay recent history.

Durable
🔐

Scoped

Channels are workspace-local unless you explicitly bridge them to an external webhook. Your data never leaves your machine unless you say so.

Secure
🔀

Chainable

The output of one channel handler can publish to another channel, creating cascading multi-step pipelines without a central orchestrator.

Pipeline

✅ Agent Channels

  • Event-driven — reacts instantly
  • Loosely coupled — agents don't know about each other
  • Persistent ring buffer — replay recent history
  • Fan-out to N subscribers simultaneously
  • Optional webhook bridge to external services

⏳ Without Channels (polling)

  • Delayed — checks on a timer
  • Tightly coupled — each agent calls each other directly
  • No history — misses events if agent is offline
  • Sequential — one agent waits for another
  • Requires custom integration code per service
Channel Types

Four Kinds of Channels

Choose the right channel type for your use case. All four use the same publish/subscribe API — they differ only in delivery semantics.

📢

Broadcast Channel

Every subscriber receives every message. Use for notifications, status updates, and dashboards. No acknowledgement required.

Example topics
system.ready · alerts.info · logs.debug default type
📬

Queue Channel

Messages are delivered to exactly one subscriber (round-robin). Perfect for distributing work across a pool of worker agents so no item is processed twice.

Example topics
jobs.process · emails.send · tasks.queue work distribution
📋

Request/Reply Channel

Publisher sends a query and blocks (or awaits) for a single reply from exactly one subscriber. Enables synchronous-style RPC between agents.

Example topics
lookup.customer · price.check · auth.verify request/response
📅

Scheduled Channel

OpenClaw publishes a heartbeat message on a cron schedule. Any subscriber wakes up and acts — no external cron daemon or scheduler required.

Example topics
cron.daily · cron.hourly · heartbeat.ping time-driven
Quick Start

Up and Running in 4 Steps

From zero to a working event-driven pipeline in under five minutes.

1

Create a channel

Give your channel a namespaced name. Use dot notation to keep topics organised — e.g. domain.event.

Terminal
/channel create orders.new
# Output: ✅ Channel "orders.new" created (broadcast, buffer: 100)
/channel create <name> [--type broadcast|queue|rr|scheduled] [--buffer N]
2

Subscribe an agent

Tell an agent to listen on the channel. The agent will be woken up every time a new message arrives.

Terminal
/channel subscribe orders.new --agent inventory-agent
# Output: ✅ inventory-agent is now subscribed to "orders.new"
/channel subscribe <channel> --agent <agent-name>
3

Publish a message

Push a JSON payload to the channel. All subscribers receive it immediately. You can also publish from within any agent's code or skill.

Terminal
/channel publish orders.new '{"id":"ORD-4821","sku":"SHOE-42","qty":2}'
# Output: ✅ Message delivered to 1 subscriber
/channel publish <channel> <JSON payload>
4

Add an on-event handler

Wire a skill or prompt to fire automatically whenever a message arrives. The payload is injected into the skill as $event.

Terminal
/channel on-event orders.new --run "Update inventory for order $event.id, SKU $event.sku, qty $event.qty"
# From now on, every new order automatically updates inventory
/channel on-event <channel> --run "<prompt with $event placeholders>"
Reference

Full Command Reference

Every /channel command, flag, and expected output — your complete cheat sheet for Agent Channels.

Command Description Key Flags
/channel create <name> Create a new named channel --type broadcast | queue | rr | scheduled
--buffer N (messages to retain)
--cron "0 9 * * *" (for scheduled type)
/channel list List all channels in the current workspace with subscriber count and message stats --verbose show full details
/channel subscribe <channel> Subscribe the current agent (or a named agent) to a channel --agent <name>
--filter JSONPath expression to only receive matching messages
/channel unsubscribe <channel> Remove a subscription from a channel --agent <name>
/channel publish <channel> <payload> Broadcast a JSON message to all channel subscribers --priority low | normal | high
--ttl seconds before message expires
/channel on-event <channel> Register an event handler that fires on every new message --run prompt string with $event placeholders
--skill skill name to invoke
--agent which agent executes the handler
/channel off-event <channel> Remove a registered event handler --id handler ID (from on-event output)
/channel history <channel> Replay buffered messages from a channel --last N messages
--since ISO timestamp
--json output as JSON array
/channel inspect <channel> Show channel configuration, subscriber list, handler list, and buffer stats —
/channel bridge <channel> Forward all messages to an external webhook URL --url webhook endpoint
--secret HMAC signing secret
--inbound also accept POSTs from the webhook as publishes
/channel delete <channel> Delete a channel and all its history --force skip confirmation
/channel pause <channel> Temporarily stop delivering messages to subscribers (buffer still fills) —
/channel resume <channel> Resume a paused channel and flush buffered messages to subscribers —
/channel stats Show global message throughput, delivery latency, and error rates across all channels --since time window
💡

Pro tip: You can use the $event variable anywhere in your --run prompt string. Access nested fields with dot notation: $event.order.customer.email, $event.metadata.source. The full payload is also available as $event._raw.

Real-World Patterns

6 Patterns You Can Build Today

Copy any of these patterns into your own OpenClaw workspace — they're production-tested recipes, not toy demos.

🛒

E-Commerce Order Pipeline

Broadcast

When a new Shopify order arrives, fan out simultaneously to three agents: inventory deduction, Slack customer notification, and QuickBooks invoice creation.

# 1. Create the channel
/channel create shopify.orders.new --buffer 500

# 2. Wire up three independent handlers
/channel on-event shopify.orders.new --run "Deduct stock: SKU $event.sku × $event.qty"
/channel on-event shopify.orders.new --run "Post to Slack #orders: New order $event.id from $event.customer"
/channel on-event shopify.orders.new --run "Create QuickBooks invoice for $event.total to $event.customer.email"

# 3. Shopify webhook publishes via bridge
/channel bridge shopify.orders.new --inbound --url https://hooks.learnopenclaw.org/shopify
📧

Email Triage Queue

Queue

New emails publish to a queue channel. Three identical triage agents each pick up one email at a time, classify it, draft a reply, and archive it — in parallel, with no duplicate processing.

# Queue type: each message goes to exactly one subscriber
/channel create inbox.unread --type queue

/channel subscribe inbox.unread --agent triage-agent-1
/channel subscribe inbox.unread --agent triage-agent-2
/channel subscribe inbox.unread --agent triage-agent-3

# Gmail monitor publishes each unread email
/channel on-event gmail.new --run "Publish to inbox.unread: $event"
🤖

Multi-Agent Research Pipeline

Cascading

A research task spawns sub-agents via the multi-agent system. Each sub-agent publishes findings to a research.findings channel. A synthesis agent subscribes and assembles the final report once all findings arrive.

# Sub-agents publish as they complete
/channel create research.findings --buffer 50
/channel create research.complete

# Each sub-agent ends its task by publishing
# /channel publish research.findings {"topic":"market-size","data":"..."}

# Synthesis agent waits for N findings then triggers
/channel on-event research.findings --run "If research.findings has 5+ messages, synthesise all findings into a report and publish to research.complete"
⏰

Daily Briefing on Schedule

Scheduled

A scheduled channel fires every morning at 8 AM. A handler fetches your calendar, emails, Slack DMs, and GitHub PRs — and delivers a single daily briefing to your desktop.

# Scheduled type publishes automatically on cron
/channel create cron.morning --type scheduled --cron "0 8 * * 1-5"

/channel on-event cron.morning --run "Fetch today's calendar events, top 5 unread emails, open GitHub PRs assigned to me, and any Slack DMs since yesterday. Write a concise morning briefing."
🚨

CI/CD Deploy Gate

Filtered

Your CI system publishes all build events to ci.events. A filter on the subscriber ensures your deploy agent only wakes up for status: "passed" events on the main branch.

/channel create ci.events

# Deploy agent only receives passing main-branch builds
/channel subscribe ci.events --agent deploy-agent --filter "$.status=='passed' && $.branch=='main'"

# Alerting agent gets ALL failures regardless of branch
/channel subscribe ci.events --agent alert-agent --filter "$.status=='failed'"
📊

Live Dashboard Feed

Bridged

Bridge a channel to an outbound webhook so your external dashboard or Notion database receives a live feed of agent activity without any custom integration code.

# Bridge: outbound only — all messages forwarded to webhook
/channel bridge agent.activity --url https://your-dashboard.com/ingest --secret $WEBHOOK_SECRET

# Now every agent can publish to agent.activity and your dashboard updates
/channel publish agent.activity '{"event":"task_complete","agent":"research","duration_s":42}'
Best Practices

Channel Naming Conventions

A consistent naming scheme makes large multi-channel workspaces much easier to manage and debug.

📏

Convention: Use lowercase dot-separated namespaces — domain.noun.verb or domain.event. Avoid underscores and spaces. Keep names short but descriptive.

Pattern Good Examples Avoid
domain.event orders.new · user.signup · payment.failed newOrder · NEW_ORDER
domain.noun.verb github.pr.merged · shopify.inventory.low github-pr-merged · github/pr
system.scope cron.daily · cron.hourly · alerts.critical daily_cron · ALERT
agent.status agent.research.complete · agent.deploy.started researchDone · deploy
app.entity.action notion.page.created · slack.dm.received notionCreated · slackDM
🧹

Use --filter to stay focused

Rather than creating many narrow channels, create one broad channel per domain and use --filter on subscribers. This keeps your channel list manageable.

💾

Set appropriate --buffer sizes

High-volume channels (logs, metrics) should use a small buffer (10–50). Low-volume critical channels (orders, alerts) should use larger buffers (500–1000) so you never miss a message.

🚧

Pause before deployments

Use /channel pause before updating a handler, then /channel resume afterward. Buffered messages will flush in order — no events lost during updates.
