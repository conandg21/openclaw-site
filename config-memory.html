⚙️ memory { }

Memory Configuration

The memory block controls where your agent stores memories, how long it keeps them, how it searches them, and when it auto-summarises daily notes.

JSON Schema

Full memory { } Block

Here's the complete memory configuration with inline annotations explaining each field:

{
  "memory": {
    "directory":       "~/.openclaw/memory",  // Where all memory files live
    "memoryFile":      "MEMORY.md",           // Long-term curated memory filename
    "dailyNotes":      true,                  // Create daily note files (YYYY-MM-DD.md)
    "dailyNotesDir":   "memory/",             // Subdirectory for daily notes
    "retentionDays":   90,                // Auto-delete daily notes older than this (0 = keep forever)
    "maxMemoryWords":  2000,                // Warn when MEMORY.md exceeds this
    "autoSummarise":   false,                // Auto-condense old daily notes into MEMORY.md
    "autoSummariseAfter": 30,                 // Days before auto-summarise kicks in
    "search": {
      "enabled":       true,
      "fuzzy":         true,                  // Allow approximate matches
      "maxResults":    10,                    // Max results from /memory search
      "indexOnStartup": true                  // Build search index at launch
    },
    "backup": {
      "enabled":       false,
      "destination":   "",                    // Path or cloud URL to back up to
      "frequency":     "daily"                // "hourly" | "daily" | "weekly"
    },
    "encryption": {
      "enabled":       false,
      "keyFile":       "~/.openclaw/memory.key"
    }
  }
}
Top-Level Fields

Field Reference

All configurable fields in the memory block, with types, defaults, and descriptions.

Field Type Default Description
directory string ~/.openclaw/memory Root directory where all memory files (MEMORY.md, daily notes, search index) are stored. Use ~ for home directory.
memoryFile string MEMORY.md Filename for the long-term curated memory file. Always stored in the root of directory.
dailyNotes boolean true Enable automatic creation of daily note files (one per day, named YYYY-MM-DD.md).
dailyNotesDir string memory/ Subdirectory where daily notes are stored (relative to directory).
retentionDays number 90 Auto-delete daily notes older than this many days. Set to 0 to keep all notes forever.
maxMemoryWords number 2000 Warn when MEMORY.md exceeds this word count. Helps maintain curated, concise memory.
autoSummarise boolean false Enable automatic summarisation of old daily notes into MEMORY.md. Uses LLM tokens.
autoSummariseAfter number 30 Days old a daily note must be before it becomes eligible for auto-summarisation.
How It Works

Memory Two-Tier Architecture

OpenClaw uses two complementary storage layers:

📝

Daily Notes

One file per day (memory/2026-03-22.md). Auto-created, contains raw session logs. Subject to retention period — old notes auto-delete.

💾

MEMORY.md

Long-term curated memory. Manually edited or auto-summarised. Kept forever unless you delete it. Loaded into context at session start.

Memory Flow at Session Start

1. Load MEMORY.md → Agent has access to all curated long-term facts.

2. Load today's daily note → Agent has context of today's conversations.

3. Run agent → Agent makes decisions, uses tools, talks to you.

4. Update daily note → Session summary added at end of day.

Read the full memory guide at workspace-memory.html for strategies on organizing and maintaining your agent's knowledge.

Examples

Retention Period Examples

How retentionDays affects your daily notes storage:

retentionDays: 0

Behavior: Keep all daily notes forever. No automatic deletion.

Disk usage: Grows over time. After 1 year: ~365 notes (if one per day).

Best for: Full audit trail, complete history, research projects, or when storage is abundant.

retentionDays: 30

Behavior: Keep last month only. Notes older than 30 days auto-delete on next daily note creation.

Disk usage: Capped at ~30 notes (~500 KB per note = ~15 MB total).

Best for: Most users. Good balance of recent context without unlimited storage.

retentionDays: 90

Behavior: Keep last quarter. Notes older than 90 days auto-delete.

Disk usage: Capped at ~90 notes (~13.5 MB total).

Best for: Medium-term projects, seasonal workflows, or when you want broader context without a full year.

retentionDays: 365

Behavior: Keep full year. Notes older than 365 days auto-delete.

Disk usage: Capped at ~365 notes (~180 MB total).

Best for: Long-running projects, yearly retrospectives, or when you need rolling 12-month history.

💡

Pro tip: MEMORY.md is not affected by retention. Even with retentionDays set to 30, your curated MEMORY.md persists forever. Use auto-summarise to fold old daily notes into MEMORY.md before they expire.

Advanced Features

Auto-Summarise: Smart Memory Consolidation

Automatically distill old daily notes into your curated MEMORY.md before they expire.

How Auto-Summarise Works

1. Check age: Every session, check if any daily notes are older than autoSummariseAfter days.

2. Read old notes: Load the eligible daily note(s) into context.

3. Extract facts: Agent reads the notes and identifies key facts (decisions, insights, milestones).

4. Update MEMORY.md: Agent appends a summary to MEMORY.md with date and source note.

5. Clean up: Original daily note is deleted (or archived if backup.enabled).

Configuration

"autoSummarise": true — Enable the feature.

"autoSummariseAfter": 30 — Only summarise notes 30+ days old.

⚠️

Token cost: Auto-summarise calls your LLM to read and extract facts from old notes. On each session, if eligible notes exist, you'll spend tokens for summarisation. Plan accordingly if you have token limits or billing concerns.

Search Configuration

Memory Search Settings

Control how your agent searches memories and how many results to return.

Field Type Default Description
search.enabled boolean true Enable memory search via the /memory search command.
search.fuzzy boolean true Allow approximate/fuzzy matching. Finds "project" when searching "projects", handles typos.
search.maxResults number 10 Maximum search results to return. Higher = more context but slower search.
search.indexOnStartup boolean true Build search index when agent starts. If false, search is slower on first query.

Using Memory Search

/memory search your query here

Trigger a search to find facts from both MEMORY.md and daily notes. Results are ranked by relevance. If fuzzy is enabled, queries are forgiving of typos and near-matches.

Search Example

Query: /memory search Acme project budget

Fuzzy matches: Finds "ACME", "Akme", "projet", "budjet" if close enough.

Result: Up to 10 matching snippets from memory files, ranked by relevance.

Persistence

Backup Settings

Automatically back up memory files to a remote location.

Field Type Default Description
backup.enabled boolean false Enable automatic backups of memory files.
backup.destination string "" Where to back up: local path (e.g., ~/Backups/memory), S3 URL, Dropbox path, etc.
backup.frequency string "daily" How often to run backups: "hourly", "daily", or "weekly".

Backup Examples

Local Path Backup

"destination": "~/Backups/openclaw-memory"

Backs up to another folder on your machine. Simple, no cloud setup required.

Cloud Backup (S3)

"destination": "s3://my-bucket/memory"

Backs up to an AWS S3 bucket. Requires AWS credentials configured on your machine.

Dropbox Backup

"destination": "~/Dropbox/OpenClaw/memory"

Backs up to a Dropbox folder. Files are automatically synced to your Dropbox account and all devices.

ℹ️

Note: Backups copy files only. If encryption.enabled is false, backups are plaintext. Enable encryption if your memory contains sensitive data (passwords, health info, financial details).

Security

Encryption: Protecting Sensitive Memories

Enable AES-256 encryption for memory files if they contain sensitive information.

Field Type Default Description
encryption.enabled boolean false Enable AES-256 encryption of all memory files on disk.
encryption.keyFile string ~/.openclaw/memory.key Path to the encryption key file. Must be kept secure — losing this file means losing access to encrypted memories.

When to Enable Encryption

✓ Enable if your memory contains:

API keys, auth tokens
Passwords or passphrases
Health or medical information
Financial data (account numbers, SSNs)
Confidential business information
Personal identifiers (addresses, phone, email)

✗ Not necessary if your memory contains:

Project notes and ideas
General facts and research
Task lists and reminders
Public information
Non-sensitive conversation logs

How Encryption Works

1. Key Generation: When you enable encryption, a random AES-256 key is generated and saved to ~/.openclaw/memory.key.

2. File Encryption: All memory files (MEMORY.md, daily notes, search index) are encrypted at rest using this key.

3. Agent Access: When the agent needs to read memory, files are decrypted in memory (not on disk).

4. Backups: If backups are enabled, only encrypted files are backed up. The key itself is NOT backed up — you must manage it separately.

🔑

Critical: Back up your encryption key file separately and keep it secure. If you lose ~/.openclaw/memory.key and encryption is enabled, you will lose access to all encrypted memories permanently. There is no recovery method.
